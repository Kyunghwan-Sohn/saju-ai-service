## 1. Data Strategy Overview

### 1.1 데이터 레이어 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                     Data Strategy Layers                         │
│                                                                  │
│  Layer 1: Static Domain Data (변하지 않는 기초 데이터)            │
│  ├── 만세력 데이터 (1900~2100, 절기/간지/음양력)                  │
│  ├── 천간/지지/오행 관계 테이블 (상수)                            │
│  └── 신살 판단 규칙 테이블 (상수)                                 │
│                                                                  │
│  Layer 2: Knowledge Data (명리학 지식 데이터)                     │
│  ├── 원전 텍스트 (적천수, 자평진전, 궁통보감)                      │
│  ├── 해석 가이드라인 (격국별, 십성별, 신살별)                      │
│  └── 상담 사례 (전문가 검증 익명화 사례)                          │
│                                                                  │
│  Layer 2.5: Pre-built Content Data (사전 구축 콘텐츠) ★ NEW      │
│  ├── saju_contents: 사주 조합별 풀이 (abcllm 배치 생성)          │
│  ├── daily_fortunes: 일일 운세 (매일 abcllm 배치 갱신)           │
│  ├── compatibility: 궁합 콘텐츠 (사전 생성)                      │
│  ├── content_versions: 서술 변형 버전 관리                        │
│  └── ★ DB에서 바로 송출 (온라인 서빙 시 LLM 미호출)              │
│                                                                  │
│  Layer 3: Operational Data (운영 데이터)                          │
│  ├── 사용자 프로필 & 사주 데이터                                  │
│  ├── 상담 이력 & 메시지                                          │
│  ├── 운세 콘텐츠 조회 로그                                       │
│  └── 결제 & 구독 데이터                                          │
│                                                                  │
│  Layer 4: Analytics Data (분석 데이터)                            │
│  ├── 사용자 행동 로그 (이벤트 트래킹)                             │
│  ├── AI 응답 품질 평가 데이터                                     │
│  ├── 프롬프트 성능 메트릭                                         │
│  └── A/B 테스트 결과                                              │
│                                                                  │
│  Layer 5: Training Data (학습 데이터 -- v2.0+)                    │
│  ├── 전문가 검증 해석 데이터                                      │
│  ├── 사용자 피드백 기반 선호 데이터                                │
│  └── 파인튜닝용 학습 데이터셋                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 데이터 수집-활용 사이클

```
수집 (Collect)          가공 (Process)         활용 (Utilize)          평가 (Evaluate)
─────────────          ─────────────         ─────────────          ─────────────
원전 디지털화    →     청크 분할/태깅   →     abcllm 배치 입력  →     콘텐츠 품질
만세력 데이터    →     정규화/검증      →     사주 산출         →     산출 정확도
abcllm 배치     →     품질검증/필터    →     DB 저장(즉시송출) →     사용자 피드백 ★NEW
사용자 입력      →     암호화/익명화    →     개인화 상담       →     CSAT 점수
상담 로그        →     패턴 분석        →     프롬프트 개선     →     응답 품질
피드백 수집      →     라벨링           →     파인튜닝 (v2.0+) →     전문가 평가
```

### 2.2 원전 데이터 가공 파이프라인

```
Step 1: 디지털화
  - 원전 텍스트 확보 (공개 도메인 한문 원전 + 한글 번역본)
  - OCR 또는 수동 입력으로 디지털 텍스트 확보
  - 한문 원전과 한글 해설 병렬 구성

Step 2: 구조화
  - 장/절/항 단위로 구조화
  - 원문(한문) + 직역 + 의역 + 현대 해설 4단 구조

Step 3: 메타데이터 태깅
  - 주제 분류: 천간론 / 지지론 / 격국론 / 용신론 / 신살론 / 통변론
  - 관련 요소: 해당되는 천간/지지/오행/십성/신살 태깅
  - 활용 상황: 어떤 사주 특성에 적용되는 지식인지 태깅

Step 4: 품질 검증
  - 명리학 전문가 1인 이상 검수
  - 번역/해석 오류 교정
  - 현대적 맥락 부적합 내용 플래그
```

### 4.2 청크 분할 전략

| 전략 | 설명 |
|------|------|
| **청크 크기** | 500자 (한국어 기준). 명리학 원전의 한 개념이 대체로 300~600자에 해당. |
| **오버랩** | 100자. 문맥 유실 방지. |
| **분할 우선순위** | 1) 단락 경계, 2) 문장 경계, 3) 글자 수 기준. 의미 단위를 보존. |
| **특수 처리** | 한문 원전은 원문 + 해설을 하나의 청크로 묶어 컨텍스트 유지. |

### 4.3 검색 전략

```python
async def retrieve_knowledge(
    user_query: str,
    saju_context: SajuContext
) -> list[KnowledgeChunk]:
    """
    하이브리드 검색: 벡터 유사도 + 메타데이터 필터
    """

    # 1. 검색 쿼리 구성 (사용자 질문 + 사주 키워드 보강)
    enriched_query = f"{user_query} {saju_context.geokguk} {' '.join(saju_context.special_stars)}"

    # 2. 벡터 유사도 검색 (cosine similarity)
    vector_results = await pgvector_search(
        query_embedding=embed(enriched_query),
        top_k=10
    )

    # 3. 메타데이터 필터 (관련 오행/십성/주제)
    filtered = filter_by_metadata(
        results=vector_results,
        elements=saju_context.dominant_elements,
        topic=saju_context.topic
    )

    # 4. 리랭킹 (사주 데이터와의 관련성)
    reranked = rerank_by_relevance(filtered, saju_context)

    # 5. 상위 5개 반환
    return reranked[:5]
```

## 5. User Data & Feedback Collection

### 5.1 수집 데이터 분류

| 데이터 유형 | 수집 항목 | 목적 | 보관 기간 |
|-------------|----------|------|-----------|
| **프로필 데이터** | 생년월일시, 성별, 양/음력 | 사주 산출 | 회원 탈퇴 시 삭제 |
| **상담 데이터** | 사용자 질문, AI 응답, 주제 | 대화 이력, 품질 분석 | 90일 후 익명화 옵션 |
| **행동 데이터** | 페이지뷰, 클릭, 세션 시간 | UX 개선, A/B 테스트 | 1년 |
| **피드백 데이터** | 응답 만족도, 정확도 평가 | AI 품질 개선 | 영구 (익명화) |
| **결제 데이터** | 구독/결제 이력 | 비즈니스 분석 | 5년 (법적 보관) |

### 5.2 사용자 피드백 수집 체계

```
│  │                  │                                        │
│  │  - 상담 완료율 (중간 이탈 vs 완료)                         │
│  │  - 후속 질문 빈도 (관심도 = 더 많은 질문)                  │
│  │  - 재방문 패턴 (동일 주제 반복 상담)                       │
│  │  - 세션 시간 (더 긴 세션 = 더 높은 몰입)                  │
│  │  - 공유 행동 (SNS 공유 = 높은 만족)                       │
│  └─────────────────┘                                         │
│                                                              │
│  ┌─────────────────┐                                         │
│  │ Explicit Signal  │  사용자가 직접 제공하는 피드백           │
│  │                  │                                        │
│  │  - 응답별 좋아요/싫어요 (매 AI 응답에 제공)                │
│  │  - CSAT 설문 (상담 종료 시, 5점 척도)                     │
│  │  - 자유 텍스트 피드백 (선택적)                              │
│  │  - "정확하지 않아요" 신고 버튼                              │
│  └─────────────────┘                                         │
│                                                              │
│  ┌─────────────────┐                                         │
│  │ Expert Review    │  전문가 정기 평가                       │
│  │                  │                                        │
│  │  - 분기별 100개 응답 샘플링                                │
│  │  - 명리학 전문가 3인 평가                                  │
│  │  - 정확성/적절성/근거제시 3축 평가                          │
│  │  - 평가 결과 → 프롬프트 개선 반영                          │
│  └─────────────────┘                                         │
└─────────────────────────────────────────────────────────────┘
```

### 6.4 비용 모니터링

```python
# LLM 비용 추적
class LLMCostTracker:
    """
    PRD 제약: MVP 기간 월 500만원 인프라 (LLM ~300만원)
    일일 한도: ~10만원
    """

    async def track_usage(self, response_metadata):
        daily_cost = await redis.incr_float(
            f"llm_cost:{today()}",
            amount=self._calculate_cost(
                input_tokens=response_metadata.input_tokens,
                output_tokens=response_metadata.output_tokens,
                provider=response_metadata.provider
            )
        )

        if daily_cost > DAILY_LIMIT * 0.8:  # 80% 경고
            await alert("LLM daily cost approaching limit", daily_cost)

        if daily_cost > DAILY_LIMIT:  # 한도 초과
            await alert("LLM daily cost exceeded!", daily_cost)
            # 신규 상담 일시 중단 또는 응답 토큰 제한
```

## 8. Data Governance & Privacy

### 8.1 데이터 분류 및 보호 수준

| 데이터 등급 | 예시 | 보호 조치 |
|-------------|------|-----------|
| **Critical (극비)** | 생년월일시, 상담 내용 | AES-256 암호화, 접근 로그, 최소 권한 |
| **Sensitive (민감)** | 이메일, 닉네임 | 암호화 저장, 마스킹 |
| **Internal (내부)** | 사주 산출 결과, 운세 | 접근 제어 |
| **Public (공개)** | 오행 설명, 명리학 해설 | 제한 없음 |

### 8.2 개인정보 처리 방침 (PIPA 준수)

| 항목 | 정책 |
|------|------|
| **수집 목적** | 사주 산출, AI 상담, 운세 제공 |
| **보유 기간** | 회원 탈퇴 시 30일 유예 후 삭제 |
| **암호화** | 생년월일시 AES-256-GCM, DB 연결 TLS, 전송 HTTPS |
| **익명화** | 상담 내역 90일 후 자동 익명화 옵션 |
| **접근 권한** | 개발자 DB 직접 접근 금지, Admin API를 통한 간접 접근만 허용 |
| **감사 로그** | 개인정보 조회/수정/삭제 시 로그 기록 |
| **제3자 제공** | LLM API 호출 시 개인정보 제거 후 전송 (생년월일시는 사주 결과로 변환 후 전송) |
| **삭제 요청** | 사용자 요청 시 72시간 내 완전 삭제 처리 |

### 8.3 LLM API 전송 시 개인정보 보호

```
중요: LLM API에 원본 생년월일시를 절대 전송하지 않는다.

사용자 데이터:
  이름: 김지은
  생년월일: 1998-05-15 14:30 (양력)

      ↓ [사주 엔진: 서버 내부 계산]

LLM에 전송되는 데이터:
  사주: 무인(戊寅) 정사(丁巳) 갑오(甲午) 신미(辛未)
  일간: 갑목(甲木)
  오행: 목2 화3 토1 금2 수0
  격국: 상관격
  신살: 역마살, 천을귀인

  → 이름, 생년월일시, 나이 등 개인 식별 정보 없음
  → 사주 결과만으로는 개인 식별 불가
```

### 8.4 데이터 생명주기

```
생성 (Create)
  └── 사용자 가입, 프로필 생성, 상담

활용 (Active Use)
  └── 사주 산출, AI 상담, 운세 생성

보관 (Retention)
  ├── 상담 내역: 90일 후 익명화 옵션
  ├── 행동 로그: 1년 보관
  └── 결제 정보: 5년 법적 보관

삭제 (Deletion)
  ├── 회원 탈퇴: 30일 유예 → 완전 삭제
  ├── 삭제 요청: 72시간 내 처리
  └── 익명화 데이터: AI 학습용으로 영구 보관 (동의 시)
```
