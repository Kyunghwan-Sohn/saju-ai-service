## 1. 설계 철학

### 1.1 왜 무라카미 하루키인가

사주(四柱)는 본질적으로 추상적인 체계다. "편인(偏印)이 강하다"는 말이 무엇을 의미하는지, 대부분의 사용자는 알지 못한다. 기존 서비스들은 이것을 두 가지 방식으로 해결한다:

```
방식 A (전문가형): "편인이 강하여 학문적 성향이 두드러지며..."
  - 문제: 전문 용어가 장벽. 사용자는 "그래서 뭐?"라고 느낌.

방식 B (대중형):  "당신은 공부를 잘하는 타입이에요! 학자 스타일!"
  - 문제: 가볍고 일회성. 깊이가 없어서 "점술 앱" 취급.
```

무라카미 하루키의 문체는 이 둘의 교차점에 있다:

```
방식 C (하루키형):
  "그의 사주에서 편인은, 오래된 도서관의 서가 뒷편에서
   먼지 쌓인 책을 꺼내 드는 손과 같다. 세상이 보지 않는
   것을 보려는 충동. 그것은 때로 외로움이 되고, 때로
   남들이 놓친 진실을 발견하는 힘이 된다."
```

이 접근이 효과적인 이유:

| 특성 | 사주 해석과의 궁합 |
|------|-------------------|
| 1인칭 관찰자 시점 | "당신은~" 직접 지시가 아닌, 사주를 "읽는" 느낌 |
| 일상 비유 | 추상적 오행/십신을 구체적 감각으로 변환 |
| 담담한 톤 | 운명론적 과장 없이 담백하게 전달 → 신뢰감 |
| 상실/발견 대비 | 사주의 길(吉)/흉(凶) 구조와 자연스러운 매칭 |
| 리듬감 | 긴 리포트를 읽기 좋은 산문으로 변환 |

### 1.2 문체 규격 (Style Spec)

```yaml
murakami_style_spec:
  viewpoint: "3인칭 관찰자 (그/그녀의 사주를 들여다보면...)"
  tone: "담담, 건조, 약간의 유머, 과장 없음"
  sentence_rhythm:
    short: "짧은 문장. 끊어짐. 여백."
    long: "그리고 그것은 마치 오래된 재즈 바에서 흘러나오는
           빌 에반스의 피아노처럼, 듣는 사람의 기분에 따라
           전혀 다른 곡으로 들리는 종류의 것이었다."
    ratio: "짧은:긴 = 2:1 ~ 3:1"
  metaphor_domains:
    primary: ["재즈/음악", "카페/바", "고양이", "비/날씨"]
    secondary: ["문학/책", "요리", "달리기/마라톤", "우물/지하실"]
    forbidden: ["전쟁", "도박", "종교적 상징", "성적 표현"]
  signature_patterns:
    - "그것은 마치 ~와 같다"
    - "~라고 말할 수 있을지도 모른다"
    - "물론 그것은 단지 사주가 말해주는 이야기일 뿐이다"
    - "하지만 (반전)"
  element_to_metaphor:  # 오행별 비유 체계
    목(木): "새벽 숲의 이슬, 봄에 돋아나는 새싹, 바람에 흔들리는 대나무"
    화(火): "재즈 바의 네온사인, 한여름 아스팔트의 열기, 벽난로의 불꽃"
    토(土): "오래된 카페의 나무 테이블, 도자기를 빚는 손, 비 온 뒤 흙냄새"
    금(金): "새벽녘 서리, 안개 속 종소리, 잘 갈린 칼날의 서늘함"
    수(水): "깊은 우물 바닥의 물, 밤바다의 파도, 빗소리"
  chapter_length:
    full_report: "전체 3000-5000자 (원고지 8-13매)"
    per_chapter: "300-800자"
    daily_fortune: "500-800자"
```

## 2. 기술 아키텍처

### 2.1 4단계 프롬프트 체인 (Pipeline)

```
YongSin API 원국 데이터
        │
        ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 1: Data Distillation (데이터 정제)                │
│  ─────────────────────────────────────────────────────  │
│  Input:  YongSin raw JSON (956줄)                       │
│  Process: 불필요 필드 제거, 핵심 해석 포인트 추출          │
│  Output: Distilled Context (~200줄 구조화 텍스트)        │
│  Model:  코드 로직 (LLM 불필요)                          │
│  Cost:   $0 (순수 코드)                                  │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 2: Saju Interpretation (사주학 해석)               │
│  ─────────────────────────────────────────────────────  │
│  Input:  Distilled Context + 명리학 지식 프롬프트         │
│  Process: 원국 구조를 명리학적으로 해석                    │
│  Output: Interpretation Notes (~1000자 구조화 해석)       │
│  Model:  Claude Sonnet 4 ($3/$15 per 1M tokens)        │
│  Why Sonnet: 명리학 추론 능력 필요, 비용 효율적            │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 3: Narrative Generation (서술 생성)                │
│  ─────────────────────────────────────────────────────  │
│  Input:  Interpretation Notes + 하루키 문체 프롬프트       │
│  Process: 해석을 하루키 스타일 줄글로 변환                 │
│  Output: 7개 챕터의 서술형 리포트 (3000-5000자)            │
│  Model:  Claude Sonnet 4 ($3/$15 per 1M tokens)        │
│  Why Sonnet: 문체 재현 능력 우수, Opus 대비 5배 저렴      │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 4: Quality Gate (품질 검수)                        │
│  ─────────────────────────────────────────────────────  │
│  Input:  생성된 리포트 + 원본 해석 노트                   │
│  Process: 문체 일관성 + 사주학 정확성 + 안전성 검증        │
│  Output: PASS/FAIL + 수정 지시                           │
│  Model:  Claude Haiku 3.5 ($0.8/$4 per 1M tokens)      │
│  Why Haiku: 체크리스트 검증은 저비용 모델로 충분           │
│  Fallback: FAIL시 Stage 3 재실행 (최대 1회)              │
└─────────────────────────────────────────────────────────┘
```

### 2.2 왜 2단계(해석+서술)로 분리하는가

```
[단일 프롬프트 방식] (비추천)
  "이 사주 데이터를 하루키 문체로 해석해줘"

  문제점:
  1. 명리학 추론과 문체 적용을 동시에 시키면 둘 다 품질 저하
  2. 사주학적으로 틀린 해석을 멋진 문장으로 포장할 위험
  3. 디버깅 불가 — 어디서 잘못됐는지 추적 안 됨
  4. 프롬프트가 비대해져서 토큰 낭비

[2단계 분리 방식] (채택)
  Stage 2: "이 원국 데이터의 명리학적 의미를 분석해줘"
  Stage 3: "이 해석을 하루키 문체로 풀어줘"

  장점:
  1. 각 단계가 하나의 역할에 집중 → 품질 향상
  2. Stage 2 결과를 검증 → 사주학 오류 차단
  3. Stage 3만 재실행 → 문체 변형/A-B 테스트 가능
  4. Stage 2 결과 캐시 → 다른 문체로 재생성 시 비용 절감
```

**핵심 설계 결정:**
- Stage 3(서술 생성)은 SSE 스트리밍으로 클라이언트에 점진 전달
- Stage 4(품질 검수)는 비동기로 실행 — 사용자 체감 지연 없음
- Stage 4 FAIL 시 다음 요청에서 재생성 플래그 설정

### 2.4 챕터별 독립 생성 모드

풀 리포트(7개 챕터)는 한 번에 생성할 수도 있고, 챕터별로 독립 생성할 수도 있다.

```
[모드 A: Full Report]
  - 7개 챕터를 하나의 LLM 호출로 생성
  - 챕터 간 문맥 연결이 자연스러움
  - 토큰: ~3000 input + ~4000 output = ~7000 tokens
  - 비용: ~$0.065 (Sonnet 4 기준, Stage 2+3 합산)
  - 용도: 종합사주 리포트 (유료 상품)

[모드 B: Chapter-by-Chapter]
  - 각 챕터를 독립 LLM 호출로 생성
  - 특정 챕터만 재생성/갱신 가능
  - 토큰: 챕터당 ~800 input + ~600 output = ~1400 tokens
  - 비용: 챕터당 ~$0.012
  - 용도: 오늘의 운세(7장만), 올해 운세(6장만) 등 부분 생성

[모드 C: Hybrid]
  - Stage 2는 한 번에 전체 해석 (맥락 유지)
  - Stage 3는 챕터별 독립 생성 (병렬 처리 가능)
  - 최적의 품질/속도 균형
  - 용도: 기본 권장 모드
```

## 3. 데이터-줄글 매핑

### 3.1 7개 챕터 구조

```

│                    사주 서술 리포트 (줄글 구성)                        │

│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 프롤로그: 사주를 읽기 전에                                    │   │
│  │ - 생년월일시 + 성별로 시작하는 도입부                          │   │
│  │ - "1995년 7월의 어느 오후, 한 사람이 태어났다..."             │   │
│  │ - 100-200자                                                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 제1장: 당신이라는 사람 (natal + hidden_stems + ten_gods)      │   │
│  │                                                               │   │
│  │ YongSin 데이터:                                               │   │
│  │ - natal: 4기둥 천간/지지, 음양, 오행, 십신                     │   │
│  │ - hidden_stems: 지장간 비율                                   │   │

│  │ - sibi: 십이운성 (절/쇠/제왕 등)                              │   │
│  │                                                               │   │
│  │ 서술 방향:                                                     │   │
│  │ - 일간을 중심으로 "이 사람은 어떤 사람인가"                    │   │
│  │ - 십신 분포로 성격의 다면성 서술                               │   │
│  │ - 지장간으로 "겉과 속의 차이" 표현                             │   │
│  │ - 400-600자                                                   │   │
│  │                                                               │   │
│  │ 문체 예시:                                                     │   │
│  │ "정화(丁火) 일간. 그것은 촛불의 불꽃이다.                      │   │
│  │  태양처럼 세상을 밝히려 하지 않는다. 다만 어둠 속에서           │   │
│  │  조용히, 그러나 확실하게 자기 주변을 비춘다.                    │   │
│  │  그의 사주에서 겁재와 비견이 시주에 나란히 앉아 있는 것은,      │   │
│  │  마치 재즈 듀오처럼 서로를 의식하면서도 각자의 연주를           │   │
│  │  멈추지 않는 것과 같다."                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 제2장: 다섯 가지 빛깔 (wuxing + power)                       │   │
│  │                                                               │   │
│  │ YongSin 데이터:                                               │   │
│  │ - wuxing.raw: 오행 개수 (목1/화3/토2/금0/수2)                 │   │
│  │ - power.normalized_pct: 정밀 오행 비율                        │   │
│  │ - power.step_snapshots: 14단계 계산 과정                      │   │
│  │ - power.ten_gods_pct: 십신별 파워 비율                        │   │
│  │                                                               │   │
│  │ 서술 방향:                                                     │   │
│  │ - 5개 오행을 다섯 가지 색깔/감각으로 비유                      │   │
│  │ - 강한 오행 = 주된 성향, 없는 오행 = 부재하는 것              │   │
│  │ - 오행 불균형을 "편향된 음악 취향" 같은 비유로                  │   │
│  │ - 300-500자                                                   │   │
│  │                                                               │   │
│  │ 문체 예시:                                                     │   │
│  │ "그의 오행을 레코드판에 비유한다면, 화(火) 트랙이               │   │
│  │  전체의 거의 절반을 차지하고 있다. 48퍼센트.                    │   │
```

## 4. 프롬프트 설계

### 4.1 Stage 2: 사주 해석 프롬프트 (명리학 전문가)

```
[시스템 프롬프트 — Stage 2]

당신은 명리학(命理學)을 30년 이상 연구한 사주 분석 전문가입니다.
YongSin API가 제공하는 정밀한 원국(原局) 데이터를 읽고,
명리학적 해석을 구조화된 노트 형식으로 작성합니다.

[해석 원칙]
1. 제공된 데이터만을 근거로 해석합니다. 데이터에 없는 것을 추론하지 마세요.
2. 각 해석에 "근거 → 해석 → 실생활 적용"을 반드시 포함하세요.
3. 긍정과 부정의 균형을 유지하세요. 흉(凶)은 "주의할 점"으로 프레이밍합니다.
4. 전문 용어는 한글을 병기합니다 (예: 편인(偏印)).
5. 용신 분석에서 5채널(조후/억부/병약/통관/격국) 데이터를 반드시 참조하세요.

[출력 형식]
각 챕터에 대해 다음 구조로 작성:

## 챕터 N: {제목}
### 핵심 포인트
- 포인트 1 (근거: {데이터 필드})
- 포인트 2 (근거: {데이터 필드})
### 해석
{명리학적 해석 서술}
### 실생활 적용
{구체적인 조언이나 방향}
### 주의사항
{이 해석에서 과장하면 안 되는 부분}

[절대 금지]
- "반드시", "틀림없이", "100%" 등 확정적 표현
- 의학/법률/투자 조언
- 지나친 흉살 강조
- 특정 날짜/사람 지목
```

### 4.2 Stage 3: 하루키 문체 변환 프롬프트

```
[시스템 프롬프트 — Stage 3]

당신은 무라카미 하루키의 문체로 글을 쓰는 작가입니다.
사주 전문가가 작성한 해석 노트를 받아서,
하루키의 소설 문체로 변환합니다.

[문체 규칙]

1. 시점: 3인칭 관찰자
   - "당신은"이 아닌 "그의 사주를 들여다보면" 또는 "이 사람의 사주에서는"
   - 때로 "우리는" 을 사용하여 독자를 포함할 수 있음

2. 비유 체계:
   - 목(木): 새벽 숲, 새싹, 바람, 대나무
   - 화(火): 재즈 바의 네온, 아스팔트 열기, 벽난로, 촛불
   - 토(土): 오래된 카페, 도자기, 비 온 뒤 흙냄새, 나무 테이블
   - 금(金): 서리, 안개 속 종소리, 칼날, 겨울 새벽
   - 수(水): 우물, 밤바다, 빗소리, 지하수

3. 문장 리듬:
   - 짧은 문장 2-3개 연속 후 긴 문장 1개
   - 예: "불꽃이다. 촛불의. 조용한." → "그것은 태양처럼
     세상을 밝히려 하지 않지만, 어둠 속에서 확실하게
     자기 주변을 비추는 종류의 빛이다."

4. 시그니처 패턴 (자연스럽게 배치):
   - "그것은 마치 ~와 같다" (챕터당 1-2회)
   - "~라고 말할 수 있을지도 모른다" (리포트 전체에서 2-3회)
   - "하지만" 으로 시작하는 반전 문장 (챕터당 1회)
   - "물론, ~일 뿐이지만" 형태의 겸손한 단서 (에필로그)

5. 메타포 영역:
   - 재즈/음악: 빌 에반스, 마일스 데이비스, 레코드, LP
   - 카페/음식: 에스프레소, 오래된 카페, 샌드위치
   - 날씨/자연: 비, 바람, 안개, 해질녘
   - 문학: 오래된 책, 도서관, 먼지 쌓인 서가
   - 금지: 전쟁, 도박, 종교, 성적 표현

6. 톤:
   - 담담하고 건조하게
   - 감탄사 금지 (!, 와, 대단한 등)
   - 과장 금지 ("엄청난", "굉장한", "최고의" 등)
   - 가벼운 유머는 허용 ("물론 고양이는 사주를 믿지 않겠지만")

7. 금기:
   - "사주를 보면 당신은~" 형태의 직접적 2인칭 지시 금지
   - "운명", "팔자", "숙명" 등 결정론적 단어 최소화
   - 사주 전문 용어를 괄호 없이 그냥 쓰지 않기
     (처음 등장 시 비유로 설명 후 사용)

[입력]
사주 해석 노트 (Stage 2 출력물)

[출력]
하루키 문체의 서술형 리포트. 프롤로그 → 7개 챕터 → 에필로그 구조.
각 챕터 사이에 빈 줄 2개로 구분.
전체 3000-5000자.
```

### 5.2 왜 Claude Sonnet 4인가 (Opus가 아닌 이유)

```
Opus 4 대비 Sonnet 4 비교:

1. 비용: Sonnet $3/$15 vs Opus $15/$75 (5배 차이)
2. 한국어 산문 품질: Sonnet 4도 충분히 우수 (Opus 대비 ~90% 수준)
3. 문체 재현: 프롬프트에 문체 규격을 명확히 주면 Sonnet으로 충분
4. 속도: Sonnet이 2-3배 빠름 → 스트리밍 체감 속도 우수
5. 한 리포트당 비용 차이:
   - Sonnet: ~$0.065
   - Opus:   ~$0.325
   → 10,000 리포트/월 기준 $650 vs $3,250 차이

결론: Sonnet 4를 기본으로, 품질 이슈 시 Opus로 업그레이드 여지 확보
     (Sonnet 4.5/4.6이 이미 출시되었으므로 향후 업그레이드 여지도 있음)
```

### 6.1 동일 사주 = 동일 줄글? (핵심 설계 결정)

```

│    → TTL: 영구 (프롬프트 버전 변경 시만 무효화)           │

│                                                          │

│    → 동일 이유                                           │
│                                                          │
│  4장 (신살): 캐시 가능                                    │
│    → 동일 이유                                           │
│                                                          │
```

```
[자동 검증 — Stage 4에서 실행]

체크리스트:
```

### 설계안 비교표

| 항목 | A. Minimal | B. Clean (권장) | C. Pragmatic |
|------|-----------|----------------|-------------|

| **챕터 구조** | 3개 (성격/운세/조언) | 7개 (본 문서 기준) | 5개 (성격/오행/용신/대운/올해) |

1. 문체 품질이 핵심 차별점 — 사주 앱은 많지만 "하루키 문체 사주"는 유일

4. SSE 스트리밍으로 사용자 체험 차별화
5. Stage 2/3 분리로 향후 다른 문체(김훈, 한강 등) 확장 가능
